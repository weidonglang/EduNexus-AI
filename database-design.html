一、总体数据库设计

项目数据库使用 MySQL，表结构由 Flyway 管理。Flyway 的好处是每一次数据库结构变化都有版本文件记录，别人启动项目时可以自动建表，避免手动执行一堆 SQL。

答辩讲法： 我们的数据层不是简单堆表，而是按“用户权限域、教务业务域、通知审计域、扩展信息域”拆分。 主链路是 sys_user → student → course_selection → course_offering → course， 这条线能解释学生登录、选课、课表、考试和成绩查询的大部分业务。

四、核心业务链路如何落到数据库

登录与权限 sys_user 校验账号密码 → sys_user_role 找角色 → sys_role_menu 找菜单 → sys_role_permission 找接口权限。
学生选课 student 找当前学生 → course_offering 找可选教学班 → course_selection 写入选课记录。
抢课防超卖 Redis 先扣 selection:offering:{offeringId}:remaining → 数据库再写 course_selection → 写库失败则回滚 Redis 库存。
成绩录入 教师或管理员选择学生和课程 → 写入或更新 academic_grade → locked 控制是否允许修改。
考试安排 管理员维护 exam_schedule → 学生通过自己的 course_selection 找到相关考试。
通知公告 管理员发布 notice → 系统为用户生成 user_notification → 前端展示已读/未读。
学籍申请 学生提交 student_status_change_application → 上传材料写入 status_change_attachment → 管理员审核。

五、Redis 键值对设计

Redis 不替代 MySQL，MySQL 是最终数据源。Redis 在项目中用于高并发抢课、请求幂等、短锁和状态监控。

负责人Redis KeyValueTTL/生命周期作用关联数据库表
魏语石 selection:offering:{offeringId}:remaining 整数，表示教学班剩余名额 有过期时间；管理员修改教学班后主动删除 抢课库存缓存、并发扣减、满员快速判断、防超卖、失败回滚。 course_offering、course_selection
郭凤圣 selection:request:{requestId} 请求处理状态或处理结果 短期保存，避免重复提交 防止同一次抢课请求因为重复点击或网络重试被执行多次。 course_selection
敖东磊 selection:grab:lock:{offeringId}:{username} 锁标记，通常是处理中状态 很短 TTL，例如几秒 限制同一学生短时间内重复抢同一教学班，避免重复进入核心逻辑。 sys_user、student、course_offering
Redis 和数据库的一致性说明： 抢课时 Redis 负责快速扣减库存，数据库负责最终保存选课记录。 如果 Redis 扣减成功但数据库写入失败，系统会把 Redis 库存加回去。 如果 Redis 不可用，系统可以退回数据库兜底逻辑，功能不崩溃，只是性能会下降。
老师问“为什么不用 Redis 直接当数据库？”时可以这样答： Redis 只适合做高并发下的快速判断和临时状态控制，最终选课结果必须落到 MySQL 的 course_selection 表，因为 MySQL 有外键、唯一约束、事务和持久化能力。 所以本项目采用“Redis 抗并发，MySQL 保结果”的设计。

六、Redis 知识从零讲解与成员应用

这一页适合答辩时从零解释 Redis：先讲 Redis 是什么，再讲为什么项目需要 Redis，最后按三个人的分工说明 Redis 知识是怎么落到代码和业务里的。

30 秒开场讲法： Redis 是一个基于内存的 key-value 数据库，读写速度比普通数据库查询更快，适合保存临时状态、热点数据、库存计数、短期锁和请求幂等标记。 在我们的项目里，MySQL 负责最终数据，Redis 负责高并发抢课过程中的快速判断和临时控制。

1. Redis 是什么

Redis 可以理解为一个很快的“内存字典”。普通数据库保存的是表、行、列； Redis 保存的是 key -> value。

key-value 内存读写 高性能

2. Redis 为什么快

Redis 主要把数据放在内存中，适合频繁读写的小数据。 抢课时每个请求如果都查 MySQL，数据库压力会很大；先查 Redis 可以更快判断库存和重复请求。

内存 减少数据库压力

3. Redis 不是万能的

Redis 适合做临时状态和加速层，但最终选课结果仍然要写入 MySQL。 因为 MySQL 有事务、外键、唯一约束和持久化能力。

MySQL 保结果 Redis 抗并发

Redis 基础概念

概念通俗解释本项目里的例子答辩说法
key Redis 里的名字，类似字典的索引。 selection:offering:1:remaining 我们用有规律的 key 表示某个教学班的剩余名额。
value key 对应保存的值。 82 库存 key 的 value 是剩余座位数。
TTL 过期时间，到时间后 Redis 自动删除。 短锁 3 秒或 5 秒后自动释放。 短锁不能一直存在，所以必须设置较短过期时间。
INCR/DECR 对数字做加一或减一操作。 抢课时扣减课程库存。 高并发抢课时先用 Redis 原子扣减库存，减少数据库压力。
SETNX 只有 key 不存在时才设置成功。 设置短锁、设置请求处理中标记。 用 SETNX 可以判断是否已经有人或同一个请求正在处理。
缓存击穿/未命中 Redis 里没有数据，需要回数据库查询。 库存 key 不存在时，根据 MySQL 容量和已选人数重新计算。 Redis 没命中时会回到数据库计算，再把结果写回 Redis。
降级 Redis 不可用时，不让系统直接崩溃。 Redis 连接失败时走数据库兜底。 Redis 是加速层，不是唯一链路，所以系统可以退回 MySQL。

Redis 常见数据结构怎么理解

Redis 不只有字符串，但本项目为了保持课程设计清晰，主要使用最容易讲明白的 String 类型。 下面这些结构是 Redis 的基础知识，答辩时可以说明“我们当前主要用 String，其他结构属于可扩展方向”。

数据结构通俗理解适合场景本项目是否使用
String 一个 key 对应一个简单值，可以是数字、文本、JSON 字符串。 库存数量、短锁标记、请求处理结果。 已使用。库存、幂等、短锁都可以用 String 表达。
Hash 一个 key 下面有多个字段，类似一行对象。 缓存用户资料、课程详情、教学班详情。 可扩展。以后可以缓存教学班详情。
List 有顺序的列表，可以从左边或右边插入弹出。 消息队列、任务队列、最近操作记录。 当前未使用。
Set 无重复集合。 保存某课程已抢到的学生集合、去重集合。 可扩展。当前去重主要由 MySQL 唯一键和 Redis 幂等 key 完成。
Sorted Set 带分数排序的集合。 排行榜、优先级队列。 当前未使用。

本项目为什么选择 String 类型

库存是数字

selection:offering:{offeringId}:remaining 的值就是剩余名额，例如 82。 数字可以直接用 Redis 的 decrement/increment 进行扣减和回滚。

短锁是标记

selection:grab:lock:{offeringId}:{username} 只需要表示“这个学生正在处理这个课程请求”，值可以很简单，关键是必须设置 TTL。

幂等是结果

selection:request:{requestId} 可以保存处理中状态，也可以保存处理结果。 同一个 requestId 再来时，就能知道是否已经处理过。

Redis 命令和项目代码的对应关系

Redis 操作项目里对应的业务为什么这样用讲解重点
GET 读取库存 key、读取请求结果、读取短锁状态。 判断 Redis 中是否已有缓存或状态。 先读 Redis，命中就可以少访问数据库。
SET 写入库存值、写入请求处理结果。 把数据库计算出的临时结果放入 Redis。 Redis 保存的是加速数据，不是最终选课结果。
SETNX 设置短锁、设置 requestId 处理中标记。 只有 key 不存在时才成功，适合防重复处理。 同一学生重复点击时，第二次设置会失败，从而被拦截。
DECR 抢课时扣减教学班库存。 Redis 对单个数字 key 的扣减是原子操作，适合高并发库存。 库存小于 0 就说明已经满员，直接返回 FULL。
INCR 数据库写入失败时回滚库存。 Redis 扣了库存但 MySQL 没写成功，需要补回来。 这就是缓存和数据库一致性的处理。
EXPIRE / TTL 控制短锁和幂等 key 的过期时间。 临时状态不能永久存在。 短锁 TTL 很短；幂等结果可以稍微长一点。
DEL 管理员修改教学班后清理库存 key。 课程容量变了，旧缓存就不能继续用。 删除后下次抢课会重新按 MySQL 计算库存。
PING Redis 状态监控页检测连接。 确认 Redis 是否可用。 答辩现场可以展示 PONG 或连接正常。

Redis 与 MySQL 的职责边界

问题Redis 负责MySQL 负责为什么要分开
课程是否还有名额 快速扣减剩余库存。 保存课程容量和最终选课记录。 Redis 快，MySQL 稳。
请求是否重复 保存 requestId 处理状态。 通过唯一键防止重复选课。 Redis 提前拦截，MySQL 最终兜底。
学生是否正在重复提交 短锁限制几秒内重复进入。 保存用户、学生和选课记录。 短锁是临时控制，不应该写入数据库。
系统是否可恢复 Redis 断开时可以放弃加速。 保留完整业务数据。 Redis 不应成为系统唯一数据源。

Redis 抢课完整时序

1. 学生点击抢课 前端生成 requestId，请求带上教学班 ID 和登录 token。
2. 后端检查幂等 key 检查 selection:request:{requestId} 是否已经存在；存在就说明这个请求处理过或正在处理。
3. 设置学生短锁 尝试写入 selection:grab:lock:{offeringId}:{username}，并设置几秒 TTL，防止短时间重复点击。
4. 读取或预热库存 如果 selection:offering:{offeringId}:remaining 不存在，就根据 course_offering.capacity 和 course_selection 已选人数计算。
5. Redis 扣减库存 对库存 key 做扣减。如果结果小于 0，说明已经满员，回补并返回 FULL。
6. MySQL 最终落库 写入 course_selection。如果写入失败，说明数据库约束或容量校验没通过，需要回滚 Redis 库存。
7. 缓存请求结果 把本次 requestId 的处理结果写入 Redis，后续同一个 requestId 再来时直接返回。

三个人分别怎么使用 Redis 知识

魏语石：库存 key 与原子扣减

魏语石负责把 Redis 用在抢课库存控制上。每个教学班都有一个库存 key： selection:offering:{offeringId}:remaining。 例如 selection:offering:1:remaining 表示教学班 1 的剩余名额。

用到知识：key-value 用到知识：数字扣减 用到知识：缓存回滚
讲解口径： 抢课时系统先扣 Redis 库存，如果扣完小于 0，说明课程满员，直接返回 FULL。 如果 Redis 扣减成功但 MySQL 写入失败，就把 Redis 库存加回来，保证缓存和数据库不矛盾。

对应数据库：course_offering.capacity 表示容量， course_selection 表示已选记录。

郭凤圣：requestId 与请求幂等

郭凤圣负责把 Redis 用在重复提交控制上。前端每次抢课生成一个唯一的 requestId，后端用它组成 Redis key： selection:request:{requestId}。

用到知识：SETNX 用到知识：幂等 用到知识：TTL
讲解口径： 用户可能因为网络卡顿连续点击抢课。如果每次点击都进数据库，就可能造成重复处理。 所以我们把 requestId 写入 Redis，同一个 requestId 再次提交时，系统直接返回已有处理结果。

对应前端：选课页面生成 requestId。 对应后端：抢课服务判断请求是否已经处理。

敖东磊：短锁、TTL 与运行监控

敖东磊负责把 Redis 用在短锁和运行状态展示上。 短锁 key 是 selection:grab:lock:{offeringId}:{username}， 用来限制同一学生短时间重复抢同一门课。

用到知识：短锁 用到知识：TTL 用到知识：PING/key 扫描
讲解口径： 短锁不是为了长期锁住用户，而是为了拦住几秒内的重复提交。 Redis 监控页还能显示连接状态、PING 结果、key 数量、TTL 和缓存值，方便答辩现场证明 Redis 真的在工作。

对应页面：Redis 状态监控页、压测历史报告页、Python 压测面板。

从零讲 Redis 到项目落地的完整流程

第一步：理解 key-value Redis 用 key 找 value。本项目用 selection:offering:1:remaining 找到教学班 1 的剩余名额。
第二步：理解 TTL 临时状态不能永久保存，请求幂等 key 和短锁 key 都要设置过期时间。
第三步：理解并发扣减 很多学生同时抢课时，Redis 先扣库存，库存不足就快速拒绝。
第四步：理解最终一致 Redis 扣库存只是第一步，最终结果必须写入 MySQL 的 course_selection。
第五步：理解降级 Redis 不通时系统不会崩溃，会退回数据库兜底，只是性能会下降。
容易被问到的问题： 如果老师问“Redis 里面的数据清空了怎么办”，可以回答： Redis 是缓存和临时状态层，库存 key 不存在时可以根据 MySQL 的教学班容量和已选人数重新计算； 最终业务数据保存在 MySQL，所以 Redis 清空不会导致选课结果丢失。

八、常见状态字段和业务含义

学生与申请状态

student.status 表示学籍状态，如在读、休学等。

student_status_change_application.status 和 student_registration_application.status 表示申请审核状态，如已提交、审核中、已通过、已驳回。

成绩状态

academic_grade.grade_status 表示成绩状态，如已发布、补考、重修、缓考。

academic_grade.locked 表示成绩是否锁定，锁定后一般不允许继续修改。

通知状态

user_notification.read_flag 表示通知是否已读。

read_at 记录用户阅读通知的时间。

反馈和论文状态

teaching_feedback.status 表示反馈处理进度。

thesis_grade.status 表示论文成绩或论文流程状态。
