开发理念

技术选择越朴素，越能看出系统设计是否扎实。

理念先于工具
AI 可以把速度拉满，但方向必须由人决定。 我们不把复杂当高级，也不把炫技当能力。能解释、能维护、能交付，才是这个项目的标准。

在今天，AI 编程能力进化得非常快。普通人学习编程的速度，很难追上 AI 生成代码能力提升的速度。 AI 可能用五分钟完成过去一天的代码工作量，这是现实，也是每个学习者必须面对的变化。

但我们也越来越清楚地看到，AI 生成的代码经常倾向于使用华丽、先进、抽象程度很高的方式完成任务。 这些写法看起来强大，却不一定适合课堂项目，也不一定方便团队成员理解、讲解和维护。

所以这个项目故意回到基础：后端使用 Spring Boot、SSM 思路和 Redis，前端使用 Vue 3 与 Element Plus。 我们不用技术堆砌证明先进，而是用已经学过、能讲清楚、能调试的技术，复刻学校教学综合信息服务平台。

三人成员分工

每个人都承担明确业务边界，并至少负责一个 Redis 落地点。

责任清晰 · 可独立讲解
组长

魏语石

负责整体架构、权限体系、后台管理、接口规范和最终集成。

  • 搭建 Spring Boot 后端结构，统一 Controller、Service、Mapper 分层。
  • 负责登录认证、角色菜单权限、管理员端用户管理和审计日志。
  • 统一 Vue 3 页面布局、路由守卫、Axios 请求封装和视觉风格。
  • 负责抢课库存缓存设计，保证高并发下不超卖。
Redis 功能：抢课库存与并发扣减

使用 selection:offering:{offeringId}:remaining 缓存教学班剩余名额，抢课时先扣 Redis 库存，数据库写入失败时回滚库存，防止超卖。

成员

郭凤圣

负责课程、选课、抢课、课表和学生端核心业务。

  • 实现课程列表、课程筛选、选课退课、容量校验和学生课表。
  • 实现选课页面分页展示，避免课程和已选记录多时页面卡顿。
  • 配合抢课接口携带 requestId，避免重复点击造成重复提交。
  • 维护学院、专业、班级、学生和课程之间的模拟数据关系。
Redis 功能：请求幂等与重复提交控制

使用 selection:request:{requestId} 记录抢课请求处理结果，同一个 requestId 再次提交时直接拦截，不重复写入数据库。

成员

敖东磊

负责教师端、成绩管理、考试管理、通知公告和文件能力。

  • 完成教师登录身份、任课课程、成绩录入修改和教学评价查看。
  • 实现成绩发布锁定、补考重修缓考状态、考试安排和考场座位信息。
  • 补充公告已读未读、文件管理、导入导出和学籍材料上传说明。
  • 维护 Redis 监控页、压测历史报告和 Python 可视化压测面板。
Redis 功能：短锁与运行状态检测

使用 selection:grab:lock:{offeringId}:{username} 控制同一学生短时间重复抢同一课程，并展示 Redis PING、TTL、key 列表和缓存清理。

技术边界

后端采用 Spring Boot 承载接口与业务逻辑，按 SSM 思路拆分 Mapper、Service、Controller。 前端采用 Vue 3、Pinia、Vue Router、Element Plus。Redis 用在抢课库存、请求幂等、短锁、缓存查看和压测对比。
